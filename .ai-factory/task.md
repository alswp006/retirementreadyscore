## ✅ F1: 입력 폼(나이/월저축/현재자산/월지출) + 저장 — 추가 AC

### **F1-AC-6: [W] disabled 버튼 클릭 무동작**
- “계산하기” 버튼이 `disabled=true`인 상태에서 사용자가 버튼을 탭/클릭해도 아래가 **모두 발생하지 않아야** 한다. (하나라도 발생하면 Fail)
  1) `/result`로 라우팅 이동  
  2) localStorage `lastInput` 저장 시도  
  3) 계산 함수 호출(결과 state 생성)

### **F1-AC-7: [W] 검증 실패 → 재입력 후 복구**
- 사용자가 어떤 필드든 범위 오류/빈 값으로 인해 버튼이 disabled인 상태에서,
  - 모든 필드를 유효 범위의 값으로 수정 완료하면 **즉시(추가 액션 없이)** 아래가 성립해야 한다. (둘 중 하나라도 아니면 Fail)
    1) 모든 오류 필드의 `hasError=false`로 전환  
    2) “계산하기” 버튼이 `disabled=false`로 전환

### **F1-AC-8: [W] 빈 입력 에러의 필드별 표시(구체화)**
- 4개 입력 중 하나라도 값이 빈 문자열(미입력)일 때, 해당 필드는 아래를 만족해야 한다. (하나라도 아니면 Fail)
  - `hasError=true`
  - `help` 텍스트가 **정확히** `필수 입력값이에요`로 노출
- 값이 입력되어 빈 상태가 해제되면 해당 필드는 **즉시** `hasError=false`가 되어야 한다. (아니면 Fail)

### **F1-AC-9: [W] 범위 오류 help 텍스트(필드별 문구 고정)**
- 범위 밖 입력 시 해당 필드는 `hasError=true`이고, help 텍스트가 아래 문구와 **정확히 일치**해야 한다. (불일치 시 Fail)

| 필드 | 조건 | help 텍스트(정확히) |
|---|---|---|
| 나이 | `< 19` | `19세 이상이어야 해요` |
| 나이 | `> 80` | `80세 이하로 입력해요` |
| 월저축 | `< 0` | `0원 이상 입력해요` |
| 월저축 | `> 10,000,000` | `1,000만원 이하로 입력해요` |
| 현재자산 | `< 0` | `0원 이상 입력해요` |
| 현재자산 | `> 10,000,000,000` | `100억원 이하로 입력해요` |
| 월지출 | `< 1` | `1원 이상 입력해요` |
| 월지출 | `> 20,000,000` | `2,000만원 이하로 입력해요` |

> 위 help 텍스트가 노출되는 동안 “계산하기”는 `disabled=true`여야 한다. (아니면 Fail)

### **F1-AC-10: [W] 정수 입력 강제(소수/문자/특수문자 처리)**
- 각 입력 TextField는 **정수만** 상태값으로 반영되어야 한다.
- 사용자가 아래 입력을 시도할 때, 내부 상태에 저장되는 값은 다음 규칙을 따라야 한다. (다르면 Fail)
  - `12.5` 입력/붙여넣기 → `12`로 저장(소수점 이하는 버림)
  - `-5` 입력/붙여넣기 → `5`로 저장(절댓값)
  - `1,000` 입력/붙여넣기 → `1000`으로 저장(구분자 제거)
  - 숫자가 전혀 없는 문자열(예: `abc`) → 빈 값으로 처리(해당 필드 `hasError=true` + help는 F1-AC-8 적용)

### **F1-AC-11: [E] localStorage 읽기/쓰기 실패 시 폴백(앱 사용 지속)**
- Home 진입 시 `lastInput`을 읽는 과정에서 예외가 발생하면:
  1) 앱이 크래시 없이 Home이 렌더링되어야 하고  
  2) 4개 필드는 모두 빈 값 상태로 표시되어야 한다. (둘 중 하나라도 아니면 Fail)
- “계산하기” 성공 후 `lastInput` 저장 과정에서 예외가 발생하면:
  1) `/result` 네비게이션은 **계속 진행**되어야 하고  
  2) Toast로 `입력값을 저장하지 못했어요`가 노출되어야 한다. (둘 중 하나라도 아니면 Fail)
- 위 모든 경우에 `console.error`가 호출되면 Fail (AC-REVIEW-2와 정합)

## ✅ F3: 결과 화면(리워드 광고 게이팅 + 결과 표시) — 추가 AC

### **F3-AC-4: [W] /result 직접 접근(state 없음) 에러 UI 문구 고정**
- `location.state`가 `null/undefined`이거나, `state.input` 또는 `state.result`가 없으면 Result는 **결과 UI 대신** 아래만 렌더링해야 한다. (하나라도 아니면 Fail)
  - Paragraph.Text: `계산 데이터를 찾을 수 없어요`
  - Paragraph.Text: `홈에서 다시 계산해 주세요`
  - Button(aria-label 포함): `홈으로` → 탭 시 `/`로 이동

### **F3-AC-5: [E] TossRewardAd 로드/표시 실패 시 fallback(결과 비공개 유지)**
- TossRewardAd가 로드 실패/에러 상태를 반환한 경우(네트워크 오류, 광고 고갈 등) 아래를 만족해야 한다. (하나라도 아니면 Fail)
  1) 상세 숫자(은퇴 가능 나이/부족 자산/추가 저축액)는 **여전히 DOM에 렌더링되지 않아야** 한다.
  2) 에러 안내 텍스트 `광고를 불러올 수 없어요`가 노출되어야 한다.
  3) Button `다시 시도`가 노출되어야 하며, 탭 시 TossRewardAd가 **재시도(재마운트/재요청)** 되어야 한다.
  4) Button `홈으로`가 노출되어야 하며, 탭 시 `/`로 이동해야 한다.

### **F3-AC-6: [U] retirementPossibleAge=null UI 문구 고정**
- `retirementPossibleAge === null`인 경우, “은퇴 가능 나이” 결과 항목 값은 숫자 대신 **정확히** 아래 문구로 표시되어야 한다. (다르면 Fail)
  - `현재 저축으로는 계산할 수 없어요`
- 이때도 “60세 기준 부족 자산”, “60세까지 월 추가 저축 필요액” 2개 항목은 기존 규칙대로 숫자(포맷 적용)로 표시되어야 한다. (아니면 Fail)

### **F3-AC-7: [E] 재계산 사이클(값 수정 → 새 결과 반영)**
- Result에서 “다시 계산하기” → Home 프리필 상태에서 **값을 1개 이상 수정** → “계산하기” → Result로 이동했을 때,
  - Result에 표시되는 3개 결과는 **수정된 입력값으로 재계산된 값**이어야 한다. (이전 결과가 그대로면 Fail)
