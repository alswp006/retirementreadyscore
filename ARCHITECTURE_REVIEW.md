# Architecture Review
Generated: 2026-04-09T07:34:41.979Z
Score: 63/100

## Summary
The codebase has a clear, minimal structure for a Toss mini-app with well-separated concerns across hooks, lib, components, and pages. However, several hooks appear unused, a critical error-handling contract is broken between useTossAd and TossRewardAd, and the isTossApp environment check is copy-pasted across four files. The overall architecture is sound but needs targeted cleanup.

## Issues
### Critical
- [consistency] useTossAd.show() silently swallows all errors internally (try/catch with only console.error, no rethrow), but TossRewardAd.tsx wraps show() in its own try/catch expecting it to throw on failure. The catch block in TossRewardAd is therefore unreachable — ad failures will never trigger the fallback unlock path via the catch branch. Only the timeout path remains as a safety net. — Files: src/hooks/useTossAd.ts, src/components/TossRewardAd.tsx / Fix: Either (a) rethrow the error inside useTossAd.show() after setting state so TossRewardAd's catch is reachable, or (b) return a discriminated result object { ok: boolean, error?: unknown } from show() and remove the try/catch in TossRewardAd, branching on the result instead. Option (a) is simpler: add 'throw e;' after console.error in useTossAd.

### Warnings
- [duplication] The isTossApp environment detection expression is copy-pasted identically across four hook files. Any change to detection logic (e.g. switching to a userAgent check) must be made in four places. — Files: src/hooks/useTossAd.ts, src/hooks/useTossLogin.ts, src/hooks/useTossPayment.ts, src/hooks/useTossPromotion.ts / Fix: Extract to a shared module, e.g. src/lib/env.ts: export const isTossApp = typeof window !== 'undefined' && !!(window as any).Toss; Then import it in all four hooks.
- [dead_code] AdSlot component is exported but not imported or used anywhere in the codebase. It appears to be a stub that was never wired up. — Files: src/components/AdSlot.tsx / Fix: Either integrate AdSlot into TossRewardAd or Result.tsx where a static ad slot is needed, or delete the file if it has no planned use.
- [dead_code] useTossLogin, useTossPayment, and useTossPromotion are fully implemented hooks with no visible import site anywhere in the codebase (App.tsx, Home.tsx, Result.tsx, or any component). They may be speculative scaffolding. — Files: src/hooks/useTossLogin.ts, src/hooks/useTossPayment.ts, src/hooks/useTossPromotion.ts / Fix: If these features are planned, add a TODO comment and move them to a feature-branch or docs/planned/ directory. If unused, delete them to reduce maintenance surface. Dead hook code is especially risky because it still runs linters, type-checks, and review cycles.
- [consistency] Result.tsx imports useTossAd directly and presumably re-implements the ad gate pattern inline, while TossRewardAd.tsx exists precisely to encapsulate that pattern (ad gate, timeout fallback, loading state, unlock logic). This creates two diverging implementations of the same flow. — Files: src/pages/Result.tsx, src/components/TossRewardAd.tsx, src/hooks/useTossAd.ts / Fix: Remove the direct useTossAd import from Result.tsx and wrap the result content in <TossRewardAd slotId="result-unlock">...</TossRewardAd> instead. This is exactly the documented usage in TossRewardAd's JSDoc.
- [consistency] src/lib/storage.ts provides a synchronous localStorage wrapper exporting getItem/setItem/removeItem, while @apps-in-toss/framework also exports async getItem/setItem/removeItem backed by Toss native storage. Home.tsx uses the synchronous localStorage version. In the Toss app environment, the native async storage should be preferred, but there is no adapter or decision point — the two APIs are used interchangeably by name. — Files: src/lib/storage.ts, src/pages/Home.tsx, src/types/apps-in-toss.d.ts / Fix: Create a unified storage abstraction in src/lib/storage.ts that branches on isTossApp: use the async Toss native storage in-app and localStorage as fallback. This also removes the naming collision between the two modules' exports.

### Suggestions
- [dead_code] Several exports appear unused within the visible codebase: cn() in utils.ts (no JSX className composition found), formatNumber() in utils.ts (only formatCurrency is imported in Result.tsx), removeItem() in storage.ts (only getItem/setItem used in Home.tsx), and the isReady value returned from useTossAd (destructured nowhere in TossRewardAd or Result.tsx). — Files: src/lib/utils.ts, src/lib/storage.ts, src/hooks/useTossAd.ts / Fix: Audit and remove unused exports, or add @internal JSDoc if intentionally kept for future use. For isReady specifically, consider whether it should gate the button render in TossRewardAd (currently the button is always shown, even if the SDK hasn't signaled readiness).
- [naming] Export style is inconsistent: all pages (Home, Result) and App use default exports, while all components (AdSlot, TossRewardAd) and hooks use named exports. This is a minor convention mismatch but causes inconsistent import syntax across the project. — Files: src/components/AdSlot.tsx, src/components/TossRewardAd.tsx, src/pages/Home.tsx, src/pages/Result.tsx / Fix: Adopt a consistent rule: named exports for everything (easier to refactor and grep), or default exports for page-level components only. Document the convention in a brief CONTRIBUTING.md or ESLint rule (import/no-default-export scoped to components/).
